# Исправление RLS политик для челленджей

## Проблема
Обнаружена бесконечная рекурсия в RLS политиках таблиц `challenges` и `challenge_participants`. Политики ссылаются друг на друга через подзапросы, что вызывает ошибку:
```
infinite recursion detected in policy for relation "challenges"
```

## Решение
Используем упрощенные RLS политики, которые разрешают всем аутентифицированным пользователям просматривать все челленджи, а фильтрацию выполняем на уровне приложения.

## Применение исправления

### Вариант 1: Через Supabase Dashboard (рекомендуется)

1. Откройте Supabase Dashboard: https://supabase.com/dashboard
2. Выберите ваш проект
3. Перейдите в SQL Editor
4. Скопируйте содержимое файла `fix-challenges-rls.sql`
5. Вставьте в SQL Editor
6. Нажмите "Run" для выполнения

### Вариант 2: Через Supabase CLI

```bash
# Убедитесь, что вы залогинены
supabase login

# Примените SQL файл
supabase db execute --file fix-challenges-rls.sql --project-ref YOUR_PROJECT_REF
```

### Вариант 3: Через psql (если есть прямой доступ)

```bash
psql "postgresql://postgres:[YOUR-PASSWORD]@[YOUR-HOST]:5432/postgres" -f fix-challenges-rls.sql
```

## Что изменилось

### Подход к безопасности:

**До (с рекурсией):**
- RLS политики пытались фильтровать данные на уровне базы данных
- Политики ссылались друг на друга через подзапросы
- PostgreSQL обнаруживал циклическую зависимость

**После (без рекурсии):**
- RLS политики разрешают всем аутентифицированным пользователям просматривать данные
- Фильтрация выполняется на уровне приложения в `challenge.service.ts`
- Пользователи могут изменять только свои собственные записи

### Новые политики:

**Challenges:**
- SELECT: Все аутентифицированные пользователи могут просматривать
- INSERT: Только если пользователь является создателем
- UPDATE: Только создатель может обновлять
- DELETE: Только создатель может удалять

**Challenge Participants:**
- SELECT: Все аутентифицированные пользователи могут просматривать
- INSERT: Только если пользователь добавляет себя
- UPDATE: Только свои записи
- DELETE: Только свои записи

### Фильтрация на уровне приложения:

Метод `getUserChallenges()` в `challenge.service.ts` теперь:
1. Получает все челленджи из базы
2. Получает записи участия пользователя
3. Фильтрует челленджи, где пользователь является создателем или участником
4. Исключает челленджи, от которых пользователь отказался

## Проверка

После применения исправления:

1. Перезагрузите страницу челленджей
2. Ошибка "infinite recursion" должна исчезнуть
3. Вы должны увидеть список челленджей или пустое состояние
4. Проверьте, что вы видите только свои челленджи и те, к которым присоединились

## Безопасность

Хотя RLS политики стали более открытыми, безопасность обеспечивается:
- Пользователи могут создавать/изменять только свои записи
- Фильтрация на уровне приложения гарантирует, что пользователи видят только релевантные данные
- Все операции записи по-прежнему защищены RLS

## Альтернативное решение (если нужна максимальная безопасность)

Если требуется более строгая безопасность на уровне БД, можно:
1. Использовать service role для операций с челленджами
2. Создать PostgreSQL функции для безопасного доступа
3. Использовать материализованные представления
